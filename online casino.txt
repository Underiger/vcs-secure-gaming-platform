## 老虎機設計（核心玩法）

這不是傳統水果盤，而是一款 Roguelite Build 構築老虎機。

### 基本規則

* 每次旋轉消耗虛擬幣。
* 三輪轉軸，每軸可包含 8~12 種符號。
* 基本符號包括：Cherry、Lemon、Bell、Bar、Lucky7、Clover、Diamond、Wild。
* 後端負責決定轉軸結果，前端只播放動畫。
* 所有 RNG 都在 Server 計算。

### Roguelite 系統

玩家可永久或暫時取得各種 Charms（護符），改變老虎機規則。所有護符效果應在伺服器端預先計算加權隨機表，並可緩存，不得即時逐符號修改機率導致效能問題。例如：

* 四葉草出現率 +30%。
* Wild 可代替任何圖示。
* 每次轉出兩個 Lucky7，第三個轉軸 Lucky7 機率增加。
* 每連續未中獎 10 次，下一次保底提高倍率。
* Diamond 中獎時額外獲得 Jackpot 點數。

玩家可以收集、裝備與搭配護符，形成不同 Build。

### 全服 Jackpot

* 每次下注的 1% 進入全服 Jackpot。獎金累積與觸發判斷應使用 Redis 原子操作，每 N 筆交易或每 10 秒批量寫入 PostgreSQL。
* 玩家有低機率進入 Jackpot 模式。
* 觸發時全服聊天室公告。
* Jackpot 中獎者獲得大部分彩金，其餘部分保留累積。
* 中獎邏輯需使用資料庫樂觀鎖（version column）避免競爭條件。
* Jackpot 相關資料需永久保存於資料庫。

### 每日與社交系統

* 每日登入獎勵。
* 每日任務。
* 今日幸運符號（每天隨機指定一種圖示提高收益）。
* 全球排行榜（今日、本週、總資產）。排行榜使用物化視圖（MATERIALIZED VIEW）每 5 分鐘更新。
* 即時聊天室。訊息需過濾 URL、限制長度與發送頻率（同 Rate Limit），避免釣魚攻擊或洗頻。
* 玩家個人資料頁與成就系統。

## Roulette 設計

* 使用歐式 Roulette（0~36）。
* 支援下注類型：

  * 單號。
  * 紅／黑。
  * 奇／偶。
  * 大／小（1-18 / 19-36）。
  * Column。
  * Dozen。
  * Split、Street、Corner（可後續擴充）。
* 後端產生結果並同步所有玩家。
* 支援多人同場觀看，每位玩家下注互不影響。
* 可建立「公共 Roulette 房」，所有人共用同一輪結果。
* 每局結束後於聊天室顯示開獎號碼與熱門下注統計。

* 玩家查詢。

## 反作弊（非常重要）

整個專案必須採用 Server Authoritative 架構：

* Client 永遠不能決定任何遊戲結果。
* Client 只能送出下注請求。
* Server 驗證餘額、扣款、計算 RNG、更新資料庫後才回傳結果。
* 所有重要操作使用資料庫 Transaction。

需要實作：

* JWT 驗證。
* Rate Limit。
* Replay Attack 防護：每個客戶端請求須攜帶 Sequence Number 與時間戳記（±5 秒有效窗），伺服器記錄 last_seq，拒絕舊封包。
* 異常下注行為偵測。
* 非法封包紀錄。
* 下注與交易日誌。
* 管理員操作日誌（Audit Log）。
* 基本反修改客戶端設計（所有權威資料皆以 Server 為準）。
* 所有下注請求必須在回合開啟後的特定有效時間內（例如 15 秒），超時拒絕。
* 傳輸加密：所有客戶端與伺服器通訊必須強制使用 TLS 1.2+（HTTPS 與 WSS），禁止明文 HTTP/WS。
* 請求簽章：客戶端對每個請求使用 HMAC-SHA256 計算簽章，簽章必須涵蓋 userId + gameType + betAmount + nonce + timestamp，防止欄位被分開篡改。登入成功後，伺服器產生隨機 HMAC 金鑰（32 位元組），透過 TLS 傳送給客戶端，並在伺服器端以 Redis 或 JWT 關聯該金鑰；伺服器驗證每次請求的簽章一致性。會話期間應定期（例如每 24 小時或登出時）輪換 HMAC 金鑰，或在 JWT Refresh 時一併更新。
* 所有遊戲結果必須使用 Node.js 內建 crypto.randomBytes() 或 WebCrypto API，禁止使用 Math.random() 或其他非密碼學安全的 PRNG。
* 所有扣款操作必須使用條件更新：UPDATE users SET balance = balance - :amount WHERE id = :userId AND balance >= :amount，並檢查受影響行數；若為 0 則拒絕交易。使用資料庫交易隔離層級（建議 READ COMMITTED 配合樂觀鎖，避免 SERIALIZABLE 的效能開銷；高風險操作可視情況升級至 SERIALIZABLE）確保一致性。

## DDoS 基礎防護

由於 Raspberry Pi 4 資源有限，為防止低頻 DDoS 或惡意 IP 耗盡連線資源，須實作以下輕量防護：

* **Nginx 速率與連線限制**（若使用 Nginx 作為反向代理）：
  - 限制單一 IP 的並發連線數 ≤ 10。
  - 限制單一 IP 的請求速率 ≤ 10 req/s，突發上限 20。
  - WebSocket 路徑（`/socket.io/`）單一 IP 連線數 ≤ 5。
* **Linux SYN Cookies**：開啟 `net.ipv4.tcp_syncookies = 1`，防止 SYN Flood 攻擊。
* **Cloudflare（可選，推薦）**：若使用網域，可將 DNS 指向 Cloudflare 免費方案，獲得全球 DDoS 清洗能力。非強制，但能在不修改程式碼的情況下顯著提升可用性。

## Raspberry Pi 4 4GB 資源最佳化指引

* PostgreSQL 設定 `shared_buffers=256MB`、`effective_cache_size=512MB`。
* Node.js 啟用 cluster 模式最多 2 個 worker。
* Socket.IO 使用 polling + websocket，限制最大連線數 200。
* 每日任務、排行榜更新等排程作業應分散於離峰時段（若需即時則使用佇列）。
* 提供 `docker-compose.arm64.yml` 確保所有映像檔支援 arm64。
